# 딜 스크리너 발표 스크립트 (June 10)

> 평가 기준: **NOI·IRR·민감도 등 핵심 분석 로직의 정확성 + 설명력**. (GUI/웹 배포는 약간의 가산점)
> 발표 방침: 코드 한 줄씩 설명 X → **기능·작업 흐름·기본 구조**(각 함수가 무엇을 계산하는지) 중심.
> 분량 ≈ 6~8분. 아래 [화면]은 데모 시연 지점.

---

## 0. 한 줄 소개 (30초)
"여러 부동산 투자 후보를 한 번에 입력받아, **NOI부터 IRR·민감도까지 정확히 계산**하고,
투자 허들 통과 여부를 판정해 **Equity IRR로 랭킹**하는 심사 보조 대시보드입니다.
일반 가치평가 툴이 '이 자산의 가치는?'을 묻는다면, 이 도구는 **'어느 딜을 살까?'**에 답합니다."

## 1. 차별점 (40초)
- 분석 단위: 단일 자산 → **다중 딜 동시 비교**
- 출력: 지표 나열 → **PASS/FAIL 판정 + IRR 랭킹 + 최대 매입가/손익분기**
- 핵심은 화려함이 아니라 **정합성** — 모든 계산이 결정론적이고 손계산으로 교차검증됩니다(테스트 107개 통과).

## 2. 구조 — 기본 설계만 (1분)
세 덩어리로 분리했습니다.
- **`engine.py` (계산 엔진):** 순수 계산만. UI를 전혀 모릅니다. → 그래서 **엑셀처럼 손계산과 1:1 대조**가 가능합니다. 정확성을 보증하는 핵심 설계.
- **`app.py` (화면):** Streamlit UI. 엔진을 **호출만** 합니다. 계산 로직을 화면에 두지 않아, UI 버그가 계산을 오염시키지 않습니다.
- **`test_engine.py` / `test_edge.py` (검증):** 손계산 정답·엣지케이스와 대조. 엔진을 고치면 반드시 먼저 통과시킵니다.

"즉 **계산(엔진) · 표시(앱) · 검증(테스트)**을 분리해, 채점 핵심인 *계산 정확성*을 독립적으로 증명할 수 있게 만든 게 설계의 요점입니다."

## 3. 핵심 계산 흐름 — 채점 핵심 (2~2.5분, 천천히)
[화면: 드릴다운 → NOI 빌드업 탭]
"흐름은 실제 언더라이팅과 같습니다."

1. **NOI 빌드업** — `noi_breakdown()`
   임대수입(GPR)에서 공실을 빼고 기타수입을 더해 **EGI**, 거기서 운영비를 빼면 **NOI**.
   이걸 보유기간 + **한 해 더(n+1년차)**까지 계산합니다. → Exit 환원에 *포워드 NOI*가 필요하기 때문(`×(1+g)` 근사 안 쓰고 명시 계산).

2. **Exit(매각) 환원** — `terminal_value()`
   매수자는 *다음 해* 수익을 보고 값을 치릅니다. **매각가 = NOI₍ₙ₊₁₎ ÷ Exit Cap**, 여기서 매각비용 차감.

3. **현금흐름 & IRR** — `unlevered_cf` / `levered_cf` / `irr`
   무차입(Project)과 차입(Equity) 현금흐름을 각각 만들고 IRR을 구합니다.
   IRR 솔버는 **Newton-Raphson + 이분법**이고, **부호변화가 1회가 아니거나 해가 없으면 '산출 불가'로 처리** — 임의값을 절대 반환하지 않습니다(다중해 방어).

4. **판정 & 랭킹** — `evaluate()`
   **Equity IRR ≥ 허들 AND 최소 DSCR ≥ 기준 AND LTV ≤ 상한**이면 PASS.
   랭킹은 유효 IRR 우선 → Equity IRR 내림차순.

[화면: NPV↔IRR 일관성 언급] "할인율을 허들로 둔 **Equity NPV가 0이 되는 지점이 곧 IRR=허들**이라, 두 지표가 항상 일관됩니다."

## 4. 데모 시나리오 (2분)
[화면: 비교 테이블]
- 프리셋 3딜(프라임 오피스 / 물류센터 / 임대주택)이 동일 기준으로 비교됩니다.
- **물류센터만 PASS**, 나머지는 허들 미달 FAIL — 사유까지 표시.
- **음의 레버리지** 짚기: "프라임 오피스는 캡(3.9%)이 대출금리(4.5%)보다 낮아, 빌려서 사면 오히려 손해 — 그래서 Equity IRR이 Project IRR보다 낮습니다. 2022~23년 서울 프라임 오피스가 실제 이랬습니다."

[화면: 드릴다운]
- **현금흐름 바차트 · 1D/2D 민감도 히트맵** — Exit Cap·금리·공실을 흔들었을 때 IRR 변화. "캡이 오를수록 IRR이 단조 감소하는 게 보입니다." (단조성도 테스트로 검증)
- **최대 매입가·손익분기 탭:** "이 딜은 **OOO억까지** 써야 허들을 맞추고, **공실 OO%·Exit Cap OO%까지** 버팁니다." → 심사역이 실제로 묻는 '얼마까지 사도 되나'에 바로 답.

## 5. 검증 (40초)
- 채권형 등 **손계산으로 정답을 아는 케이스**를 만들어 엔진과 대조(예: 무차입 채권형 IRR = 8%).
- 엣지케이스(IRR 다중해, 분할상환 잔액, 입력 검증, NPV↔IRR 일관성, 민감도 단조성)까지 **총 107개 자동 테스트 통과**.
- 설계 원칙: **"올바르게 못 만들 거면 넣지 않는다."** 정합성·검증이 기능보다 우선.

## 6. 범위와 가정 — 정직하게 (30초)
"이 도구는 **안정화된 수익형 자산**의 1차 필터입니다. 임대가 매끄럽게 성장한다고 보므로 개발·리스업·밸류애드는 범위 밖.
**세전·연 단위** 기준이고, 그 한계를 앱에도 명시했습니다. 1차 스크리닝 목적엔 충분하고, 무엇을 안 했는지를 분명히 한 게 신뢰의 일부입니다."

## 예상 Q&A
- **Q. Exit는 왜 수익환원법만?** → 1차 스크리너는 운영 중인 수익자산이 대상이라 직접환원이 표준. (주거 분양전환·거래사례 등은 자산 유형이 달라 별도 모델 영역.)
- **Q. 캡은 NOI 기준인데 IRR은 취득비 포함 원가 기준 — 모순 아닌가?** → 캡은 시장 호가 관례(가격 분모), IRR은 실제 투입원가 대비 수익률. 각자 맞고 서로 일관.
- **Q. 왜 어떤 딜은 IRR '산출 불가'인가?** → 현금흐름 부호변화가 여러 번이면 IRR이 복수일 수 있어, 임의값 대신 무효 처리하고 랭킹 최하위로(보수적 설계).
- **Q. IRR만으로 줄 세우면 규모를 무시하지 않나?** → 맞습니다. 그래서 **거르기는 IRR·DSCR·LTV 다중 기준**, 그리고 NPV·Equity Multiple을 같은 화면에 함께 보여 규모·가치까지 판단합니다.
- **Q. 세금은?** → 세전 기준(범위 밖). 모든 딜에 동일 적용되어 비교·랭킹엔 영향 없음.
